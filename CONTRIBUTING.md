# Contributing to Ritoras

This document describes how to contribute to **Ritoras**, the open-source iOS
keyboard extension that records audio and sends it to a self-hosted Whisper
transcription server over Tailscale. It adapts the conventions of the Linux
kernel's contributor documentation to a GitHub pull-request workflow. The
intended reader is someone who has already read `README.md` and understands what
Ritoras does — this document is about *how* to contribute, not *what* the
project is.

---

## Obtain the current source tree

Clone the repository and regenerate the Xcode project from its single source of
truth:

```shell
git clone https://github.com/<you>/ritoras.git
cd ritoras/
xcodegen generate
```

**Never edit or commit `Ritoras.xcodeproj`.** That directory is generated by
XcodeGen from `project.yml` and is listed in `.gitignore`. If you need to add a
file, a framework, or a build setting, modify `project.yml` and regenerate.
(See `.gitignore` — `*.xcodeproj`, `xcuserdata/`, `DerivedData/`, and `build/`
are all excluded. Your branch should never touch them.)

**Rationale:** XcodeGen `project.yml` is a declarative, diffable, reviewable
specification. Hand-editing `.xcodeproj` bundles produces XML diffs that are
impossible to review meaningfully and frequently causes merge conflicts.

## Understand the hard constraints

Ritoras's keyboard extension runs under a **48 MB Jetsam memory cap** imposed by
iOS on third-party keyboard extensions. This is not a preference or a
performance target — it is a hard platform limit. If the keyboard extension's
resident memory exceeds this threshold, the operating system terminates it
without notice.

This single constraint drives several architectural decisions you must respect:

- **UIKit in the keyboard, not SwiftUI.** SwiftUI's first-render overhead costs
  ~5–10 MB of additional resident memory inside the 48 MB budget. UIKit
  `UIInputViewController` is the baseline. Contributions that port the keyboard
  view to SwiftUI will be rejected on memory-budget grounds.

- **No in-process inference models.** The SymSpell prediction engine (used for
  inline suggestions) has a measured baseline of approximately **25 MB resident
  memory** — see `RitorasTests/SymSpellMemorySpike.swift`. This is budgeted.
  Adding a second in-memory model or index that pushes the total over 48 MB
  will be rejected.

- **Stream-to-file recording.** `keyboard/Sources/AudioRecorder.swift` uses
  `AVAudioRecorder` which writes directly to a temp file. Never buffer a full
  audio clip in RAM.

- **No third-party HTTP libraries in the keyboard.** `keyboard/Sources/WhisperClient.swift`
  uses `URLSession` directly — no Alamofire, no Moya. Every kilobyte of static
  linker output competes with the Jetsam budget.

If your change touches the keyboard extension or `shared/` (which is compiled
into the keyboard), run `SymSpellMemorySpike.swift` as part of your validation.
Changes that inflate resident memory beyond the tested baseline will not be
merged.

**Rationale:** The 48 MB Jetsam cap is an iOS kernel parameter for keyboard
extensions (`jetsam_priority` on the `com.apple.keyboard-service` process).
There is no override, no entitlement, and no appeal. The project treats this
as a hard design constraint from the start rather than discovering it after
a feature is built.

## Describe your changes

Every pull request needs a description that answers three questions, in this
order:

1. **What problem does this solve?** — Start with the motivation. Describe the
   user-visible impact or the bug being fixed. If you are adding a feature,
   explain why it belongs inside the 48 MB budget.
2. **How does it work?** — A short description of the implementation approach.
   This does not need to be a line-by-line walkthrough, but it should let a
   reviewer know what to expect before reading the diff.
3. **What changes were required?** — Which files were touched and why. If you
   touched files outside your subsystem (e.g., a keyboard change that required
   touching `project.yml`), say so explicitly.

**Rationale:** The kernel's canonical patch format puts the "why" before the
"what" because implementation details change during review but the motivation
does not. A reviewer who understands the problem first can evaluate the
solution more quickly. This applies to PR descriptions the same way it applies
to commit messages — see "Format your commit messages" below for the full
spec.

---

## Separate your changes

One logical change per pull request. Do not mix refactors with features with
bug fixes.

**Good:**
- One PR that refactors `keyboard/Sources/AudioSession.swift` to extract a
  configuration struct, *or* one PR that adds a new feature — not both.
- A fix for a Jetsam memory regression is its own PR, not bundled inside a
  feature branch.

**Bad:**
- A single PR that renames half the symbols in `shared/`, adds a new
  transcription-language picker, and fixes a typo in `docs/SIDeload.md`.

If you have a refactor that touches many files to prepare for a feature, submit
the refactor first, wait for it to be reviewed and merged, then submit the
feature on top. This is not bureaucracy — it makes reviewing possible.

**Rationale:** When a single PR contains multiple logical changes, a reviewer
cannot approve one part without accepting another they may not have fully
reviewed. Separate PRs let each change be reviewed, tested, and reverted
independently.

## Check your code

Build and test before opening a pull request.

### Build

The authoritative build command — the one CI runs — is:

```shell
xcodegen generate && \
  xcodebuild \
    -project Ritoras.xcodeproj \
    -scheme Ritoras \
    -destination 'generic/platform=iOS' \
    -configuration Release \
    build \
    CODE_SIGNING_ALLOWED=NO \
    CODE_SIGNING_REQUIRED=NO \
    CODE_SIGN_IDENTITY=""
```

If this does not produce `** BUILD SUCCEEDED **`, your PR is not ready.

### Test

```shell
xcodebuild test -scheme RitorasTests \
  -destination 'platform=iOS Simulator,name=iPhone 15,OS=17.0'
```

The test target contains **8 test files** in `RitorasTests/` covering the
SymSpell prediction engine, bigram predictor, learned words store, Apple spell
checker provider, and the memory-spike baseline test. If you touch the
prediction or spell-check path, add a corresponding test.

### Style

The project does not use SwiftLint or SwiftFormat. Code follows standard Swift
5.9 conventions:

- Four spaces for indentation (no tabs).
- Meaningful identifier names — avoid abbreviations that are not domain-standard
  (e.g., `vc` for `viewController` is fine; `txt` for `transcript` is not).
- `MARK:` comments for organizing large files (the existing files use them).
- No force-unwrapping outside test assertions (no `!` in production code unless
  the compiler requires it for an IBOutlet or similar).

When in doubt, match the style of the file you are editing. If you are creating
a new file, read two existing similar files first and follow their conventions.

## Format your commit messages

Every commit message in Ritoras follows the **canonical commit format**,
adapted from the Linux kernel convention to GitHub-native conventions.

### Subject line

```
subsystem: concise summary of the change
```

The `subsystem` is one of the following (use the one that best describes the
scope of the change):

| Subsystem | Covers |
|-----------|--------|
| `keyboard` | `keyboard/` — the UIKit keyboard extension, audio recording, Whisper client |
| `app` | `app/` — the SwiftUI container app (settings, onboarding) |
| `shared` | `shared/` — build-time config, shared types |
| `tests` | `RitorasTests/` — any test file |
| `scripts` | `scripts/` — deployment and utility scripts |
| `docs` | `docs/` — any documentation file |
| `build` | `project.yml`, `ExportOptions.plist` — the build system and XcodeGen spec |
| `ci` | `.github/workflows/` — CI workflow changes |

The summary should not be a filename — it should concisely describe the change.

**Good:**
```
keyboard: cap PCM buffer ring to stay under Jetsam limit
```

**Less good (reads like a filename, not a description):**
```
keyboard: update AudioRecorder.swift
```

### Body

The body is required for any non-trivial change. Line-wrap at **75 columns**.

Write the body in enough detail that someone reading it six months later can
understand *why* the change was made, not just *what* it does. Start with the
motivation and problem description, then describe the implementation.

```
keyboard: switch to .default AVAudioSession mode

The .spokenAudio mode is implicated in OSStatus 561145187
(AVAudioSessionErrorCodeCannotStartRecording) on iOS 17.x when
the keyboard extension is re-shown after a previous recording
session. The .default mode avoids this path while preserving
the same recording quality for our 16 kHz mono AAC use case.

Switch AudioSession.swift to .default and retest the record →
stop → re-record loop. Also deactivate the session on stop
with notifyOthersOnDeactivation: false to avoid interfering
with the system keyboard's audio session.
```

### Backtraces and logs

If your commit references a crash log, diagnostic output, or struct dump, trim
it to the relevant lines and remove timestamps, register values, and session-
specific identifiers. The reader needs to see the *pattern*, not the raw dump.

**Good (excerpt):**
```
Thread 0 name:  com.apple.main-thread
Thread 0 Crashed:
0   RitorasKeyboard        0x1043a8000 specialized AudioRecorder.start() + 112
1   RitorasKeyboard        0x1043a8400 KeyboardViewController.micTapped() + 44
```

**Not good (full unredacted dump):**
```
2025-07-17 14:32:01.123456+0300 RitorasKeyboard[12345:67890] ...
```

### Separator

If your commit message includes non-changelog content (e.g., notes about what
changed between patch versions, acknowledgments, request for specific review
attention), separate it from the changelog body with a single line containing
only three dashes (`---`):

```
keyboard: cap PCM buffer ring to stay under Jetsam limit

The PCM buffer ring in AudioRecorder grew without bound during
long recordings, causing resident memory to climb past the 48 MB
Jetsam cap after approximately 90 seconds of continuous recording.

Cap the ring at 16 buffers and reuse the oldest buffer instead of
allocating new ones. Measured resident memory after this change
stabilizes at ~32 MB during a 5-minute recording.

---
v2: reduced cap from 32 to 16 after reviewer feedback
```

### Trailer tags

The following trailers appear at the end of commit messages, each on its own
line, after a blank line separating them from the body:

`Co-authored-by: Name <email>`
:  Someone else materially contributed to this change. Adding this tag asserts
   that the named person co-authored the work and consents to its inclusion
   under the project's MIT license. Every Co-authored-by line becomes a
   co-author on GitHub.

`Fixes: #123`
:  This commit closes GitHub issue #123. Adding this tag asserts that the
   change resolves the issue described there. GitHub auto-closes the issue
   when the commit lands on `main`.

`Reported-by: Name <email>`
:  Someone reported the problem this commit fixes, but did not contribute code.
   Adding this tag credits them without implying they endorse the fix.


`Reviewed-by: Name <email>`
:  Someone reviewed the code and approves of the change. Adding this tag
   asserts that a maintainer or invited reviewer has signed off on this commit
   during review. **Do not add this tag without the named person's explicit
   consent.** The project adds this during merge; you do not need to add it
   yourself.

Example with trailers:

```
keyboard: handle AVAudioSession interruption during recording

When a phone call arrives during recording, AVAudioSession fires an
interruption notification. Without handling it, the recording stops
but the UI stays in "recording" state until the user manually taps
stop.

Register for .AVAudioSessionInterruptionNotification in
AudioSession.swift and bridge it to the KeyboardViewController
state machine so the mic button returns to idle on interruption.

---
v2: moved interruption handling into AudioSession per review

Fixes: #42
Reported-by: Alex <alex@example.com>
```

## Submit your pull request

Pull requests are submitted against the `main` branch. There are no release
branches, no develop branch — `main` is the single integration branch.

1. Push your branch to your fork (or a branch on the upstream repository if you
   have write access).
2. Open a pull request against `main`.
3. The PR title should follow the same `subsystem: summary` convention as the
   commit subject line. If your PR contains multiple commits, the PR title
   should describe the overall change, not just the first commit's subject.
4. The PR description should follow the guidance in "Describe your changes"
   above — motivation before implementation.
5. CI runs automatically (`.github/workflows/build.yml` on `macos-15`). Wait
   for a green build. If CI fails, fix it before requesting review.

**Do not force-push during review unless the reviewer asks you to.** The review
trail of incremental commits is valuable — it shows how the change evolved in
response to feedback. When the PR is ready to merge, the maintainer may squash
or they may merge with all commits; either way, the discussion history is
preserved.

---

## Respond to review

When a reviewer leaves comments:

- Address every comment. If you agree, make the change in a new commit and
  reply "Done." If you disagree, explain your reasoning in a reply — do not
  silently ignore the comment.
- Push new commits to the same branch as you address feedback. Do not amend or
  rebase existing commits during review (unless the reviewer explicitly asks
  for it).
- If you are asked to split a PR into smaller PRs, do it. It is not a rejection
  of your code — it means the change is too large to review as a single unit.

**Don't get discouraged — or impatient.** Review is a dialogue. If a reviewer
asks for changes, it is because they believe the change is worth landing but
needs refinement. If a reviewer rejects a change on memory-budget grounds, it
is because the 48 MB limit is not negotiable. This is not personal; it is
physics.

If a reviewer does not respond within a reasonable time (a few days), a polite
ping on the PR is fine. The project is maintained by individuals with their own
schedules.

## Pre-submission checklist

Here are some basic things that contributors **should** do if they want their
pull requests reviewed and merged quickly. These are all **above and beyond**
the guidance above.

### (a) Provide documentation

1. Every new user-visible behavior is reflected in `README.md` or the
   appropriate file in `docs/`. If your PR adds a configuration option, update
   the settings description. If it changes the server protocol, update
   `docs/SERVER-CONTRACT.md`.

2. Every architectural change touching the `app/`/`keyboard`/`shared` boundary
   (adding a new shared type, changing how config is passed, rearranging the
   target dependencies in `project.yml`) updates `docs/IMPLEMENTATION-PLAN.md`
   if it affects the phase roadmap or the directory structure diagram.

3. All new public types and functions in `shared/` carry doc comments. Private
   and internal declarations do not require them, but a brief comment on
   non-obvious logic is appreciated.

### (b) Check your code with the available tooling

4. `xcodegen generate` runs cleanly from a clean checkout — no warnings, no
   errors, and the regenerated `Ritoras.xcodeproj` is byte-identical (you did
   not hand-edit the `.xcodeproj`).

5. The project builds without warnings:
   `xcodebuild -project Ritoras.xcodeproj -scheme Ritoras -destination 'generic/platform=iOS' -configuration Release build CODE_SIGNING_ALLOWED=NO`.
   Warnings that existed before your change should not be introduced by your
   change.

6. You have not added `Ritoras.xcodeproj`, `xcuserdata/`, `DerivedData/`,
   `build/`, `*.ipa`, or any other generated artifact to your commit. These are
   listed in `.gitignore` — double-check with `git status --short` before
   committing.

### (c) Test your code

7. `xcodebuild test -scheme RitorasTests` passes against an iOS Simulator
   destination. All 8 existing test files must pass.

8. If you touched the keyboard extension or `shared/` (which is compiled into
   the keyboard target), `RitorasTests/SymSpellMemorySpike.swift` still passes.
   Verify that your change did not inflate the keyboard extension's resident
   memory — the test asserts a **≤25 MB baseline** for the SymSpell index
   alone. If the total keyboard extension budget is 48 MB and SymSpell takes
   25, there is very little headroom for leaks.

9. If you added a new prediction, spell-check, or word-processing path, you
   added a corresponding test in `RitorasTests/`. Untested prediction code will
   not be merged.

10. If you touched the recording or transcription path (`keyboard/Sources/AudioSession.swift`,
    `keyboard/Sources/AudioRecorder.swift`, `keyboard/Sources/WhisperClient.swift`),
    verify that the multipart POST contract still matches `docs/SERVER-CONTRACT.md`.
    The server expects the field named `audio` (not `file`), and the response
    shape is `{"success": bool, "transcription": string}`. If the server
    contract changes, update the doc in the same PR.
